Hashing

  • Hash table is a vector of k slots, with a hashing function
  • Hash function takes a key value (ex. student number) and calculates valid bucket index
    • we want a function that generates a unique bucket index
      • key ‚Üí int between [0, k-1]
    • we lose locality for things like sorting
  • According to the Birthday Problem, collisions will be common
  • closed hashing uses linear probing to find another bucket in the table
    • aka "open addressing"
    • table is vector of K struct elements
      • allocated all at one time
      • never use push_back , basically using as fixed-size array
    • if collision, keeps adding 1 to index until an empty/zombie slot is found
      • this is known as linear probing
      • inserts take longer with more elements
    • deletion means marking bucket as zombie
      • zombie bucket is kept, but marked as undead
      • helps with navigation?
    • what happens when you filled the table?
      • create table with double the size, start at front, and re-hash
      • hash function works mod k
      • once the table is successfully set-up, no risk for malloc fail

void insertHT (HashTable& table, string name, int snum) {
	const int numBuckets = table.size();
	int slot = myhash(snum, numBuckets);
	while (ZOMBIE != table[slot] && EMPTY != table[slot].status) {
		slot = (slot + 1) % numBuckets;
	}

	table[slot].snum = snum;
	table[slot].name = name;
	table[slot].status = ACTIVE;
	numActiveElements++;
}

int lookupIndexHT(const HashTable& table, int snum) {

	const int numBuckets = table.size();
	int slot  = myhash(snum, numBuckets);
	for(int i = 0; i < numBuckets; i++) {
		if (EMPTY == table[slot].status){
			return NOT_FOUND;
		} else if (ACTIVE == table[slot].status && snum == table[slot].snum {
			return slot
		}

		slot = (slot + 1) % numBuckets;
	}

	return NOT_FOUND;
}

The Hash Function

Characteristics of a good hash function

  • Deterministic, based on key value
    • consistent value ‚Üí hash mapping
      • must always give the same answer for same input
  • Good, even spread of results over buckets
    • known as uniformity
  • Cheap to compute (should be O(1))
  • Supports a variable range
    • easy to adapt if # of buckets change
  • Hashing does not require that elements have unique keys
    • each distinct key value maps to same bucket
  • does not work well if you want to find all elements in a range

Common (and bad) hash function

Sum ASCII values of the characters, then mod by K

  • bad because you need huge number of buckets for sparse distribution
  • 1k buckets: 0% empty, avg=210, med=177, max=619
  • 10K buckets: 80% empty, avg=21, med=0, max=610
    • shortest word: "a", ASCII sum 97
    • No hash num less than 97 or more than 2621
      • very sparse
  • 100K buckets, 98% empty
    • lots of unused space

Applications

  • can use hash function to detect duplicates
  • useful for comparing "big things" to others
    • convert each big thing to something smaller using hash function, then compare smaller
    • conversion is lossy
  • hashing is often used for error correction
    • suppose downloading text file over noisy line
    • compute string hash of file at source and destination, then compare hashes
      • if they match, transfer was OK
      • if they don't data was lost
    • MD5 hash values are 32-bit int
  • plagiarism detection is a problem
    • take hash of each function (or each line after some normalization)
      • when two functions have same hash value, do detailed comparison
    • rename all variables to the same thing
    • remove all constant values (strings, numbers, etc) to the same thing
    • compute hash value for each line of code, store in a set (not a sequence)
      • order doesn't matter
    • compare each function of each student against every other one seeing if there is overlap of hash values
      • adding useless code doesn't help you
  • spell checking can be done by hashing a dictionary
  • password validation
    • before, password stored in readable plain text file /etc/passwd
  • cryptographic hash function
    • infeasible to go from hash to message
    • infeasible to modify message without changing hash
    • infeasible to find two messages with same hash
    • MD5, SHA0, SHA1 are examples of cryptographic hash functions that were broken/insecure
